# KnifeMan.exe 鼠标锁定补丁 — 实现报告

## 概述

本文档详细说明了如何通过逆向工程和二进制补丁技术，在 `KnifeMan.exe`（2002年国产武侠游戏《小李飞刀》）中实现"鼠标锁定在游戏窗口内"的功能。

**核心原理**：拦截 WndProc 的 `WM_ACTIVATE` 消息，当窗口激活时调用 Windows API `ClipCursor` 将鼠标限制在窗口客户区内，窗口失活时释放限制。

---

## 一、目标分析

### 1.1 游戏架构

| 组件 | 说明 |
|------|------|
| 编译器 | Microsoft Visual C++ 6.0 |
| 3D 引擎 | 自研引擎（thd.dll），基于 DirectX 7.0 (DirectDraw + Direct3D) |
| 输入系统 | IFC22.dll（Immersion TouchSense 力反馈鼠标）+ Win32 消息 |
| 窗口类名 | `"KnifeMan"` |
| ImageBase | `0x00400000` |

### 1.2 段布局

| 段名 | VAddr | VSize | RSize | 属性 |
|------|-------|-------|-------|------|
| .text | 0x00001000 | 0x0013BE1D | 0x0013C000 | EXECUTE\|READ |
| .rdata | 0x0013D000 | 0x0001CF8A | 0x0001D000 | READ |
| .data | 0x0015A000 | 0x0002DBE4 | 0x0000C000 | READ\|WRITE |
| .rsrc | 0x00188000 | 0x00004CE0 | 0x00005000 | READ |

**代码洞穴**：`.text` 段 RSize(0x13C000) > VSize(0x13BE1D)，末尾有 **483 字节** 的 padding 区域（RVA `0x13CE1D` ~ `0x13D000`），段加载器会用零填充这些字节。

### 1.3 已导入的关键 API

`ClipCursor` 未在原始 exe 的导入表中，但以下 API 已导入且可用：

| API | IAT RVA | 用途 |
|-----|---------|------|
| `GetModuleHandleA` | 0x13D178 | 获取 user32.dll 模块句柄 |
| `GetProcAddress` | 0x13D188 | 动态获取 ClipCursor 地址 |
| `GetClientRect` | 0x13D324 | 获取窗口客户区矩形 |
| `ClientToScreen` | 0x13D420 | 客户坐标转屏幕坐标 |
| `DefWindowProcA` | 0x13D44C | 默认窗口消息处理 |

### 1.4 关键全局变量

| 变量 | 地址 (RVA) | 说明 |
|------|-----------|------|
| HWND | 0x16C880 | 游戏主窗口句柄（BSS 中） |
| WindowObject | 0x16C830 | 游戏窗口对象指针（BSS 中） |

## 二、逆向分析过程

### 2.1 定位 WndProc

通过搜索 `DefWindowProcA` 的 `FF 15` 调用，找到游戏的主窗口过程位于 **RVA `0xF1B50`**。

关键发现：WndProc 的寄存器分配：
```
ebp = hwnd    (窗口句柄)
ebx = msg     (消息ID)
edi = wParam  (消息参数)
esi = lParam  (消息参数)
```

### 2.2 分析消息分发机制

WndProc 使用 **二级跳转表** 处理消息：

```
cmp  ebx, 0x50         ; msg 与 0x50 比较
ja  extended_handler    ; > 0x50 走扩展处理器
lea eax, [ebx-1]       ; msg - 1
cmp eax, 0xF            ; 0~15 范围
ja  default_handler

; 查表跳转
mov cl, [eax + case_table]    ; case_table 在 0xF2670
jmp [ecx*4 + jump_table]     ; jump_table 在 0xF2654
```

### 2.3 定位 WM_ACTIVATE

消息分发表分析结果：

| 消息 | msg ID | case 索引 | 原处理地址 | 说明 |
|------|--------|-----------|-----------|------|
| WM_CREATE | 0x0001 | 0 | 0x4F1CA7 | 窗口创建 |
| WM_DESTROY | 0x0002 | 1 | 0x4F1C3E | 窗口销毁 |
| WM_SIZE | 0x0005 | 2 | 0x4F2632 | 大小改变 |
| WM_SETFOCUS | 0x0007 | 3 | 0x4F1C84 | 获得焦点 |
| WM_KILLFOCUS | 0x0008 | 4 | 0x4F1C61 | 失去焦点 |
| WM_PAINT | 0x000F | 5 | 0x4F1BFD | 绘制 |
| **WM_ACTIVATE** | **0x0006** | **6** | **0x4F263E** | **激活/失活 → 默认处理** |

**核心发现**：`WM_ACTIVATE` 的 case 指数为 6，跳转表入口在文件偏移 `0xF266C`，原值为 `0x004F263E`（直接调 `DefWindowProcA` 的默认处理器）。**游戏原本没有处理窗口激活/失活消息**。

## 三、实现方案

### 3.1 方案选择

研究了四种方案后，选择 **方案A：Hook WndProc 的 WM_ACTIVATE**：

| 方案 | 原理 | 评价 |
|------|------|------|
| **A. WM_ACTIVATE Hook** | 修改消息跳转表，注入 ClipCursor 调用 | ✅ 最干净、最可靠 |
| B. SetCursorPos 轮询 | 每帧检测越界并拉回 | 光标会"跳"，体验差 |
| C. SetCapture 组合 | 鼠标捕获 + 隐藏光标 | 只在按键时有效 |
| D. DirectInput 独占 | 修改 IF22.dll 的协作层 | 可能破坏力反馈 |

### 3.2 需要修改的位置

| 修改 | 位置 | 字节 |
|------|------|------|
| `.text` 段属性 | PE 段头 +36 字节处 | 4 bytes |
| 代码洞穴 | 文件偏移 0x13CE1D | 366 bytes |
| WM_ACTIVATE 跳转 | 文件偏移 0xF266C | 4 bytes |

## 四、代码洞穴函数设计

### 4.1 内存布局

```
RVA 0x13CE1D (366 bytes):
┌──────────────────────────────────────┐
│ [0x000 ~ 0x0A6]  代码区 (167 bytes)  │
│   入口处理、初始化、锁定/解锁逻辑      │
├──────────────────────────────────────┤
│ [0x0A7 ~ 0x14F]  零填充              │
├──────────────────────────────────────┤
│ [0x150] clipCursorPtr  (4 bytes)     │  ← 运行时写入
│ [0x154] initFlag       (4 bytes)     │  ← 运行时写入
│ [0x158] "user32.dll\0" (11 bytes)    │  ← 只读
│ [0x163] "ClipCursor\0" (11 bytes)    │  ← 只读
└──────────────────────────────────────┘
```

由于代码洞穴在 `.text` 段内（默认 EXECUTE\|READ），需要修改段属性为 EXECUTE\|READ\|WRITE 以允许运行时写入 `clipCursorPtr` 和 `initFlag`。

### 4.2 完整汇编代码

```asm
; ──── 入口 (从 WndProc 的 jmp [table] 到达) ────
; ebp=hwnd, ebx=msg, edi=wParam, esi=lParam
; 栈: [esp]=saved_edi, [esp+4]=saved_esi, [esp+8]=saved_ebp,
;      [esp+0xC]=saved_ebx, [esp+0x10..0x4F]=局部变量区

push eax                  ; 保存易失寄存器
push ecx
push edx

; ──── 一次性初始化: 动态加载 ClipCursor ────
cmp dword [initFlag], 0   ; 已初始化?
jnz init_done

push strUser32            ; "user32.dll"
call [GetModuleHandleA]   ; 获取已加载的 user32.dll 句柄
test eax, eax
jz init_done              ; 失败则跳过

push strClipCursor        ; "ClipCursor"
push eax                  ; hModule
call [GetProcAddress]     ; 获取 ClipCursor 函数指针
test eax, eax
jz init_done

mov [clipCursorPtr], eax  ; 缓存指针
mov dword [initFlag], 1   ; 标记已初始化

init_done:
; ──── 分支: wParam == 0 → 解锁, wParam != 0 → 锁定 ────
movzx eax, di             ; wParam 低16位 (WA_INACTIVE=0)
test eax, eax
jz unlock_mouse

; ═══ 锁定分支 ═══
test ebp, ebp             ; HWND 有效性检查
jz unlock_mouse
cmp dword [clipCursorPtr], 0
jz handler_done

sub esp, 16               ; RECT 局部变量 (16 bytes)

; GetClientRect(hwnd, &rect)
lea edx, [esp]
push edx                  ; &rect
push ebp                  ; hwnd
call [GetClientRect]

; ClientToScreen(hwnd, &rect.left)
push esp                  ; &rect → &rect.left
push ebp
call [ClientToScreen]

; ClientToScreen(hwnd, &rect.right)
lea edx, [esp+8]          ; &rect.right
push edx
push ebp
call [ClientToScreen]

; ClipCursor(&rect)
push esp                  ; &rect
call [clipCursorPtr]      ; 通过缓存的指针调用

add esp, 16               ; 清理 RECT
jmp handler_done

; ═══ 解锁分支 ═══
unlock_mouse:
cmp dword [clipCursorPtr], 0
jz handler_done
push 0                    ; NULL
call [clipCursorPtr]      ; ClipCursor(NULL) 解除锁定

; ═══ 收尾 ═══
handler_done:
pop edx                   ; 恢复易失寄存器
pop ecx
pop eax

; ──── 默认消息处理 ────
push esi                  ; lParam
push edi                  ; wParam
push ebx                  ; msg
push ebp                  ; hwnd
call [DefWindowProcA]     ; 调用原始默认处理

; ──── WndProc 尾声 ────
pop edi                   ; 恢复保存的寄存器
pop esi
pop ebp
pop ebx
add esp, 0x40             ; 清理局部变量区
retn 0x10                 ; 返回 (4个参数 = 16 bytes)
```

### 4.3 关键设计决策

#### 为什么每次激活都重新计算 RECT？
窗口大小可能因分辨率改变、最小化/恢复等操作而变化，每次重新获取 `GetClientRect` 确保 ClipCursor 使用的矩形始终是正确的。

#### 为什么缓存 ClipCursor 指针？
`GetProcAddress` 约有 100-200 CPU 周期的开销。由于 `WM_ACTIVATE` 是低频消息（只在切换窗口时触发），性能影响极小，但缓存可以避免每次调用时的字符串查找。

#### 为什么修改 .text 段属性而不是用 .data/BSS？
- 483 字节的代码洞穴全部在 .text 段 padding 内
- 使用 .data/BSS 需要额外的地址计算和管理
- 对 2002 年的游戏来说，`.text` 段加入 WRITE 权限无安全风险

## 五、实施细节

### 5.1 PE 段属性修改

```python
# .text 段头的 Characteristics 字段位于:
# pe_offset + 4 (COFF) + 20 (COFF size) + size_opt + 36 (Characteristics)
# 原值: 0x60000020 = EXECUTE | READ | CODE
# 新值: 0xE0000020 = EXECUTE | READ | WRITE | CODE
struct.pack_into('<I', data, text_char_offset, 0xE0000020)
```

### 5.2 代码洞穴写入

```python
# .text: ROff=0x1000, VAddr=0x1000 → file_offset = RVA
cave_file_offset = 0x13CE1D
for i, b in enumerate(cave_code):
    data[cave_file_offset + i] = b
```

### 5.3 跳转表修改

```python
# 原入口: VA 0x004F263E (默认处理 → DefWindowProcA)
# 新入口: VA 0x0053CE1D (我们的代码洞穴处理函数)
JUMP_TABLE_ENTRY6_OFFSET = 0xF266C  # case 6 = WM_ACTIVATE
struct.pack_into('<I', data, JUMP_TABLE_ENTRY6_OFFSET, 0x0053CE1D)
```

### 5.4 字节码验证

全部修改已通过自动化验证，包括：
- 段属性正确设置
- 跳转表地址正确
- 代码洞穴首字节匹配预期
- 字符串数据位置正确
- `retn 0x10` 指令前有正确的 `DefWindowProcA` 调用

## 六、运行时行为

```
游戏启动
  └→ CreateWindowExA → HWND 存入 [0x56C880]
     └→ 窗口显示 → WM_ACTIVATE(wParam=WA_ACTIVE)
        └→ 跳转到代码洞穴函数
           ├→ GetModuleHandleA("user32.dll") → 成功
           ├→ GetProcAddress("ClipCursor")    → 成功
           ├→ 缓存指针, initFlag=1
           ├→ GetClientRect + ClientToScreen
           └→ ClipCursor(&rect) ← 鼠标锁定！

用户按 Alt+Tab 切出游戏
  └→ WM_ACTIVATE(wParam=WA_INACTIVE)
     └→ 跳转到代码洞穴函数
        └→ ClipCursor(NULL) ← 鼠标解锁！

用户切回游戏
  └→ WM_ACTIVATE(wParam=WA_ACTIVE)
     └→ 跳转到代码洞穴函数
        └→ GetClientRect + ClipCursor(&rect) ← 鼠标重新锁定！

游戏退出
  └→ WM_ACTIVATE(wParam=WA_INACTIVE)
     └→ ClipCursor(NULL) ← 最终解锁
```

## 八、技术亮点

1. **纯二进制补丁**：不依赖任何外部 DLL 或运行时注入，所有代码直接嵌入 exe
2. **动态 API 加载**：`ClipCursor` 未在原始导入表中，通过 `GetModuleHandleA` + `GetProcAddress` 动态获取
3. **WndProc 跳转表劫持**：精准定位消息分发表中的 `WM_ACTIVATE` 入口并重定向
4. **代码洞穴利用**：利用 PE 段对齐产生的 padding 空间（483 字节）存放完整处理逻辑
5. **栈安全**：严格遵循 WndProc 的栈帧布局，保证 `pop edi/esi/ebp/ebx; add esp, 0x40; retn 0x10` 的尾声正确执行
6. **寄存器安全**：保存/恢复所有易失寄存器（eax/ecx/edx），不破坏 WndProc 已保存的非易失寄存器
